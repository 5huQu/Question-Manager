# PDF 切题规则后台可配置化方案

## 1. 背景与问题

当前 PDF 切题器在 `server/python/src/cutter/detect_questions.py` 中通过 `AUXILIARY_MARKERS`、`NOTICE_TERMS`、`REFERENCE_FORMULA_MARKERS`、`TRAINING_MARKERS` 等 Python 常量识别章节、提示语和干扰内容。

这让基础算法简洁，但词典每增加一个词都必须修改源码、发布服务并回归验证。例如“考点精练”“专题突破”应当阻止其后的编号被识别为正式试题，却只能通过改代码实现。不同教辅模板新增题型标题时也有相同问题。

本方案将这些可变的业务词典迁移到系统后台维护，同时保留代码对切题流程、冲突处理和异常兜底的控制权。

## 2. 目标、范围与非目标

### 目标

- 让管理员可在后台新增、编辑、启停和删除切题词条，无需修改 Python 源码。
- 新规则在发布后对后续或重新执行的切题任务生效，无需重启服务。
- 保持当前默认规则和切题结果兼容；配置不可用时仍可安全切题。
- 为每次切题记录所使用的规则版本，支持问题复现、审计和回滚。
- 提供基础校验和预览，降低错误配置导致误切/漏切的概率。

### 本期范围

- 试卷和讲义 PDF 切题使用的文本规则。
- 规则的后台管理、发布、版本快照、回滚与试运行。
- Python 检测器读取规则并输出命中诊断。

### 非目标

- 不将整套切题算法改造成可视化编排或脚本引擎。
- 不在第一期开放任意 Python 表达式、脚本或不受限正则。
- 不自动重切已经人工复核过的历史批次。
- 不把 OCR 的答案/解析标记（如 `答案：`、`【解析】`）一并纳入本期；它们后续可复用同一框架。

## 3. 设计原则

1. **配置描述“匹配什么”，代码决定“如何切”**：配置只提供词条、匹配方式和类别；页面坐标、邻近上下文、题号结构、去重与裁图仍由 Python 实现。
2. **默认安全**：内置默认规则是始终可用的兜底；配置文件损坏、读取失败或校验失败时不阻断切题。
3. **先关键词，后高级规则**：第一期仅提供“包含”和“完全匹配”；确有稳定复杂模板时，再加受校验的正则模式。
4. **发布与编辑分离**：编辑中的草稿不影响生产切题；只有“保存并发布”才产生新版本。
5. **结果可追溯**：一个切题 run 固定绑定一个规则版本，不因之后规则更新而改变历史解释。

## 4. 规则模型

### 4.1 规则类别

| 类别 | 配置键 | 在检测器中的作用 | 示例 |
| --- | --- | --- | --- |
| 辅助/干扰标题 | `auxiliaryMarkers` | 进入辅助区，忽略该区编号题；遇训练/章节标题后退出 | 目录、考点精练、专题突破、思维导图 |
| 注意事项 | `noticeTerms` | 排除答题说明、考场提示等非题目文本 | 注意事项、答题、考试结束 |
| 参考公式区 | `referenceFormulaMarkers` | 避免把公式/数据区内编号识别为题号 | 参考公式、参考数据 |
| 训练/恢复标记 | `trainingMarkers` | 退出辅助区，允许后续题号识别 | 【典例训练】、一、单选题 |
| 非题目尾语 | `nonQuestionRemainders` | 排除“1. 方法总结”类编号标题 | 常见类型、方法总结 |
| 特殊章节标题 | `sectionMarkers` | 识别不符合内置章节结构的题型或训练章节 | 专项训练、题型 01、限时训练 |

`sectionMarkers` 的语义是“这是一个有效题目章节/训练章节”，不是直接产生题块。真正的题块仍由题号锚点（`1.`、`第 1 题`、`例 1` 等）产生。

### 4.2 存储格式

运行时使用 JSON，不使用 YAML。理由是现有服务已使用 `storage/config/*.json` 保存本地配置，Node 与 Python 都能原生处理 JSON，也便于 API 校验。需要批量维护时，后台可后续提供 JSON 导入/导出；不把手工编辑文件当成主路径。

文件路径：`<storageRoot>/config/pdf_slicer_rules.json`

```json
{
  "schemaVersion": 1,
  "publishedVersion": 3,
  "publishedAt": "2026-06-20T10:00:00.000Z",
  "rules": {
    "auxiliaryMarkers": [
      { "id": "aux-directory", "term": "目录", "matchMode": "contains", "enabled": true },
      { "id": "aux-focus", "term": "考点精练", "matchMode": "contains", "enabled": true },
      { "id": "aux-topic", "term": "专题突破", "matchMode": "contains", "enabled": true }
    ],
    "noticeTerms": [
      { "id": "notice-answer", "term": "答题", "matchMode": "contains", "enabled": true }
    ],
    "referenceFormulaMarkers": [],
    "trainingMarkers": [],
    "nonQuestionRemainders": [],
    "sectionMarkers": []
  }
}
```

规则条目字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 稳定 UUID/短 ID，仅用于编辑、审计和诊断 |
| `term` | 规范化后的匹配文本，长度限制 1–80 字符 |
| `matchMode` | 第一期开 `contains`、`exact` 两种 |
| `enabled` | 是否生效；禁用条目保留在历史和草稿中 |

第一期不允许单个词条跨分类复用。若确实需要，复制为两个条目，以保证意图明确。

### 4.3 优先级和冲突规则

固定优先级由程序实现，避免后台配置顺序改变核心逻辑：

1. 辅助标记命中：进入辅助区；
2. 有效章节/训练标记命中：更新章节并退出辅助区；
3. 题号锚点命中：再执行注意事项、参考公式、辅助区、缩进和尾语等排除判断；
4. 同一分类内任一启用词条命中即成立。

保存校验会阻止同一个 `term + matchMode` 同时出现在语义冲突的“辅助/干扰标题”和“训练/恢复标记”分类。其他潜在重叠仅提示，不阻止保存。

## 5. 后端与运行时设计

### 5.1 配置服务

在 `server/src/index.ts` 新增以下能力：

- `pdfSlicerRulesPath()`：定位并创建 `storage/config/pdf_slicer_rules.json`。
- `defaultPdfSlicerRules()`：返回与现有 Python 常量完全一致的默认规则。
- `normalizePdfSlicerRules()`：清理空白、去重、补 ID、限制长度、按类别校验。
- `readPdfSlicerRules()`：读取已发布规则；失败时返回内置规则并记录 warning。
- `writePdfSlicerRules()`：原子写入临时文件后替换正式文件，避免进程中断留下半个 JSON。
- `snapshotPdfSlicerRules()`：发布时写入 `storage/config/pdf_slicer_rules_history/<version>.json`，保留最近 20 个版本。

不要把规则塞进当前 `readOcrSettings()` 返回体。该接口同时承载系统设置、OCR 密钥与提示词，新增一个独立的规则资源更清晰，也更容易之后加入权限控制。

### 5.2 API

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/tools/pdf-slicer/rules` | 获取当前已发布规则及版本、默认/回退状态 |
| `PUT` | `/api/tools/pdf-slicer/rules` | 校验、保存并发布完整规则集 |
| `POST` | `/api/tools/pdf-slicer/rules/validate` | 仅校验草稿，返回错误、冲突和规范化建议 |
| `GET` | `/api/tools/pdf-slicer/rules/history` | 读取版本列表 |
| `POST` | `/api/tools/pdf-slicer/rules/rollback/:version` | 从快照恢复并发布为新版本 |
| `POST` | `/api/tools/pdf-slicer/rules/preview` | 基于给定文本模拟规则命中，见 7.3 |

当前项目未实现用户鉴权。本期可沿用“有后台访问权即可配置”的模式；接入登录/角色后，上述写接口应限定为管理员，读取接口可按产品需求放开。

### 5.3 Python 规则加载

新增 `server/python/src/cutter/rules.py`：

- 内置默认词典；
- 读取 `--rules-config` 指定文件；
- 验证顶层结构和每类条目；
- 标准化为不可变 `SlicerRules` 数据对象；
- 出错时返回默认规则并附带 `loadWarnings`。

改造 `detect_question_anchors(document, rules=None)`，将所有现有常量引用替换为 `rules` 对应字段。`run_cut.py`、`run_cut_for_question.py` 和实验服务调用处都使用同一加载函数，避免生产、命令行和实验环境规则不一致。

`run_cut_for_question.py` 增加：

```text
--rules-config /absolute/path/to/pdf_slicer_rules.json
```

`runMigratedPdfSlicer()` 在启动脚本时传入该路径。当前 run 的 `document_diagnostics_json` 写入：规则版本、规则文件哈希、是否使用默认兜底和加载告警。

### 5.4 数据库变更

在 `pdf_slicer_runs` 添加字段：

```sql
rules_version INTEGER NOT NULL DEFAULT 0,
rules_hash TEXT NOT NULL DEFAULT '',
rules_fallback_used INTEGER NOT NULL DEFAULT 0,
rules_warnings_json TEXT NOT NULL DEFAULT '[]'
```

字段也同步进入 `RunRow`、`mapRun()` 以及前端类型。切题启动前记录版本与 hash；脚本返回加载状态后更新 fallback/warnings。保留已运行批次的版本，不随新规则发布而变更。

## 6. 前端设计

### 6.1 入口与页面

在 [`SettingsPage.tsx`](/Users/imshuqu/Question/frontend/src/pages/SettingsPage.tsx) 增加“切题规则”页签，说明文字为“维护 PDF 切题的题型标题、提示词和干扰词；发布后仅影响新的或重新执行的切题任务。”

页面包含：

1. 顶部状态：当前版本、发布时间、上次发布人（本期无账号时显示“本地管理员”）、默认/回退警告。
2. 分类折叠面板：每类规则一个表格，显示词条、匹配方式、状态和删除操作。
3. 新增词条：输入词语、选择匹配方式、确认新增；默认为包含匹配和启用。
4. 草稿操作：放弃修改、校验草稿、恢复默认、保存并发布。
5. 版本历史：显示最近 20 个版本、规则摘要、查看差异、回滚。
6. 试运行：见下节。

编辑行为在浏览器本地状态中完成；只有“保存并发布”调用 `PUT`。切换设置页签时若有未保存修改，应提示确认。

### 6.2 可用性细节

- 每类规则显示命中效果提示，例如辅助标记会“暂停后续编号识别”。
- 新增前在当前分类内即时检查重复词。
- 冲突项展示具体类别和词语，不用泛化报错。
- 删除系统默认词时允许操作，但保存页提供“恢复默认规则”；也可改为默认条目只能禁用，实施时二选一并写测试。
- “保存并发布”二次确认文案明确：历史批次不会自动改变，重新切题才会使用新规则。

### 6.3 试运行

第一期采用纯文本预览：管理员粘贴一段从 PDF 复制的文本，后端按行执行与检测器相同的规则匹配，返回每行的命中分类、状态变化和被排除原因。它用于验证“考点精练”是否正确进入辅助区。

第二期可增加“选择一个已有切题批次并重新模拟”的预览。它只读取已提取的页面文本，不写 `pdf_slicer_review_items`、不生成图片，也不覆盖人工复核。

纯文本预览不是完整切题质量保证：它不覆盖 PDF 坐标、缩进和图层信息。页面要明确这一点，并在发布前推荐用一份代表性样卷重新切题复核。

## 7. 迁移与兼容策略

### 7.1 首次启动

1. 服务启动时如果规则文件不存在，按现有 Python 常量生成默认 JSON；
2. 初始版本为 `1`；
3. Python 仍保留同等默认规则，防止 Node 初始化之前的命令行执行失败；
4. 规则文件存在但不可解析时，不覆盖原文件，使用内置默认规则并在 API/日志显示告警。

### 7.2 现有规则映射

| 现有常量 | 新配置字段 |
| --- | --- |
| `AUXILIARY_MARKERS` | `auxiliaryMarkers` |
| `NOTICE_TERMS` | `noticeTerms` |
| `REFERENCE_FORMULA_MARKERS` | `referenceFormulaMarkers` |
| `TRAINING_MARKERS` | `trainingMarkers` |
| `NON_QUESTION_REMAINDERS` | `nonQuestionRemainders` |
| `SECTION_PATTERN` | 继续保留为内置结构性识别；特殊补充写入 `sectionMarkers` |

题号结构正则 `ARABIC_PATTERN`、`SECTION_QUESTION_PATTERN`、`EXAMPLE_PATTERN` 和子题排除规则不配置化，避免用户修改基础语法后造成不可预测的题块边界。

### 7.3 历史批次

- 历史 run 的 `rules_version=0` 表示“规则配置化上线前的内置规则”。
- 新版本上线不会改写历史 run，也不会触发自动重切。
- 用户点击“重新切题”时，以当前已发布规则创建新的结果，并按现有流程清理该 run 的未人工处理结果；对已人工复核 run，沿用现有确认机制或增加明确确认。

## 8. 测试与验收

### 8.1 单元测试

- 默认规则加载后的命中行为与改造前完全一致。
- 空文件、错误 JSON、未知字段、超长词、重复词与冲突词的校验。
- `contains`、`exact`、禁用词条的匹配结果。
- 辅助区进入/训练标记退出、注意事项排除、参考公式上下文排除。
- 特殊章节标记能建立有效章节上下文。
- 默认兜底能继续产生切题结果并产生 warning。

### 8.2 接口测试

- `GET` 返回默认/已发布配置和版本。
- `PUT` 对合法配置产生新版本和历史快照。
- 非法配置不写入当前版本。
- 回滚后生成递增的新版本，而不是覆盖旧版本。
- 同时保存时使用乐观锁：请求携带 `baseVersion`，版本已变化时返回 `409`，防止覆盖他人发布。

### 8.3 回归样本

至少保留三份脱敏/可提交的样本 PDF 或抽取文本夹具：

1. 标准试卷：验证原有题号与章节识别不变；
2. 含“考点精练”“专题突破”的讲义：验证辅助区编号不切题；
3. 含“参考公式”“答题注意事项”的试卷：验证编号不误切。

每个夹具断言题号列表、题块数量、排除原因和规则版本。视觉裁图仍在现有复核窗口中人工抽检。

### 8.4 验收标准

- 管理员新增“考点精练”和“专题突破”后，不修改源码、不重启服务即可生效。
- 重新切分代表样本时，其辅助区内编号不再生成题块。
- 默认配置对当前回归样本的题块数量和题号结果无变化。
- 错误配置不会使切题任务失败；运行详情能看见回退原因。
- 任一切题 run 能查询使用的规则版本和 hash。
- 可从历史版本恢复规则，并保留恢复操作的审计记录。

## 10. 发布与回滚操作手册

### 新增干扰词

1. 进入“系统设置 → 切题规则”。
2. 打开“辅助/干扰标题”，新增 `考点精练`、`专题突破`，匹配方式选“包含匹配”。
3. 点击“校验草稿”，处理重复或冲突提示。
4. 点击“保存并发布”，记下新版本号。
5. 使用试运行粘贴代表性文本，确认标题命中为“进入辅助区”。
6. 对一份代表性 PDF 发起新的或重新执行的切题，在复核窗口抽检题块。

### 新增特殊题型标题

1. 若标题只是普通的“X、单选题/填空题/解答题”，无需新增，内置章节识别已覆盖。
2. 若是“专项训练”“题型 01：函数单调性”等非标准标题，加入“特殊章节标题”。
3. 试运行确认它被识别为有效章节/训练区，而非题目本身。
4. 用样卷验证其后的 `1.`、`2.` 是否符合预期被切出。

### 回滚错误发布

1. 在“版本历史”找到上一个正常版本，查看差异后点击“回滚并发布”。
2. 系统创建新的递增版本，而不是删除错误版本；审计信息完整保留。
3. 对受影响且尚未人工复核的 run 手工重新切题；已复核批次先确认影响范围再操作。

## 11. 风险与应对

| 风险 | 应对 |
| --- | --- |
| 用户把正常题型词加入辅助分类，导致漏切 | 分类说明、试运行、发布确认、版本回滚 |
| 配置 JSON 损坏导致切题失败 | Python 内置默认规则、Node 原子写入、运行告警 |
| 规则越积越多，误命中增加 | 词条启停、版本审计、样本回归、后续支持来源规则集 |
| 多人同时编辑覆盖配置 | `baseVersion` 乐观锁和 409 冲突提示 |
| 历史结果无法解释 | run 固化版本、hash、fallback 状态和警告 |
| 任意正则带来性能/安全问题 | 第一期不开放；后续仅受限正则并做编译与复杂度保护 |

## 12. 建议的首个交付边界

首个可上线版本应完成阶段 A 和 B，但可以把版本历史、回滚和试运行放到紧随其后的阶段 C。这样先解决最核心的问题：新增“考点精练”“专题突破”等干扰词或特殊题型标记不再需要改源码；同时保留默认配置与切题结果追溯，避免把灵活性换成不可控性。
