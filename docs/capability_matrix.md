# Question Manager 当前能力矩阵

> 最后核对：2026-07-13。本文描述当前代码已经具备的能力，不是路线图。历史设计与实施过程见 `docs/tasks/`、`docs/task_run/` 和 [`import_flow_v2.md`](./import_flow_v2.md)。

## 主流程

当前主流程是：资料导入 → 整卷 OCR → 候选题解析 → 人工复核/修正 → 入库 → 题库维护与组卷 → 导出。旧 PDF 切分中心仍保留，主要用于兼容历史数据与旧工作流。

| 能力 | 状态 | 当前实现与入口 | 主要限制 |
| --- | --- | --- | --- |
| 单份或题卷/解析卷分离导入 | 已实现 | import flow v2 支持创建导入任务并关联一份或多份来源资料 | 资料角色和配对仍需用户正确选择 |
| GLM-OCR 整卷识别 | 已实现 | PDF、JPG、PNG 可启动异步识别；保存原始响应并生成 `OCRDocument` | 依赖外部 API；复杂版式仍需复核 |
| Doc2X v2 在线识别 | 已实现 | PDF 预上传、状态轮询、原始响应持久化与 `OCRDocument` 归一化 | v2 当前仅接受 PDF；不支持直接单题重跑 |
| OCR 离线 JSON 导入 | 已实现 | 可将已有响应归一化为 `OCRDocument` | 输入需符合受支持的结构 |
| 候选题解析 | 已实现 | 按题号与可配置规则生成 `QuestionCandidate`，关联题干、答案、解析、来源范围与题图 | 异常题号、跨栏和复杂混排可能产生 issue |
| 候选题编辑与复核 | 已实现 | 待确认页面使用统一可视化编辑器修改题干、结构化选项、答案和解析；支持内容版本冲突提示与本地草稿恢复 | 复杂操作的浏览器端到端测试仍不足 |
| 候选题手动修正 | 已实现 | 创建/恢复修正会话；定位原卷与答案卷；用紧凑统一编辑器调整内容，并保持区域与题图联动保存 | 尚未将选区直接送入局部 OCR；已入库候选不可在此修改，需转到正式题 |
| 单题局部 OCR + 差异确认 | 未实现 | 已有范围数据与页面渲染基础 | 需新增片段构建、provider 抽象、结果约束和 diff 确认，见 [`region_ocr_design.md`](./region_ocr_design.md) |
| 单个/批量确认入库 | 已实现 | 候选题映射到 `question_bank_items`；题库记录创建与候选状态更新使用同一事务 | 批量入库允许逐题成功/失败，界面需清晰展示部分成功结果 |
| 部分入库后重新解析 | 有限制 | 重跑会清理未入库候选草稿 | 需要版本化重解析，避免覆盖或误删已确认结果 |
| 题库维护与检索 | 已实现 | 详情、新建与待入库入口复用统一可视化编辑器；Markdown 为事实内容；支持公式、表格、结构化选项、版本冲突和本地恢复 | 批量维护能力仍可扩展 |
| 试题篮与组卷 | 已实现 | 题目加入试题篮，支持排序、分值、多份草稿和试卷级内容覆盖；可显式将单题修改同步回题库 | 复杂交互需补端到端回归 |
| PDF 排版预览 | 已实现 | 排版工作台以学生版/教师版 PDF 为唯一成品预览；试卷内容覆盖贯穿预览和最终导出，新版失败时保留上一版 | 依赖本机 LaTeX/PDF 工具链 |
| Markdown / LaTeX / PDF 导出 | 已实现 | 支持学生版/教师版及模板化导出 | PDF/LaTeX 依赖本机外部工具配置 |
| macOS / Windows 桌面打包 | 已实现 | Electron 构建与更新测试已配置 | 发布前仍需真实安装、升级与签名验收 |

## 工程保障现状

| 项目 | 状态 | 说明 |
| --- | --- | --- |
| TypeScript 构建 | 已有 | 前后端具备构建命令 |
| 后端路由、解析器与 Python 单元测试 | 已有 | 覆盖主要服务和解析逻辑；候选入库回滚与已入库候选重解析保护已有针对性回归 |
| 前端纯逻辑与组件测试 | 已有 | 已覆盖编辑内容编解码、危险粘贴清理、本地草稿、统一编辑器、版本冲突、手动修正和排版内容同步确认 |
| 浏览器端到端测试 | 不完整 | 应优先覆盖导入、OCR 状态、候选修正、入库、组卷和导出主链路 |
| PR 持续集成 | 不完整 | 应让每个 PR 至少执行构建、单元测试和核心端到端用例 |

## 维护规则

1. 功能合并时，如果能力状态或限制发生变化，同一 PR 更新本文。
2. “已实现”必须有可访问的产品入口或 API，不能只存在设计稿或未接线代码。
3. 历史任务文档不回写成现状；在顶部标注历史属性，并链接本文。
4. 路线图与技术方案分别维护，不在本表中把计划写成已完成。
