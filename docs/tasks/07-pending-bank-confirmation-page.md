# 待入库确认页规划

这份文档规划“待入库确认页”的用户体验。它接在“切题复核”和“OCR 识别”之后，目标是让用户在题目正式进入核心题库前，有一个清楚、集中、可批量处理的确认入口。

## 一句话目标

系统自动完成切题、OCR、清洗和初步检查；用户只需要判断两件事：

1. 这道题能不能直接入库。
2. 不能直接入库时，是编辑、重识别、清洗，还是删除。

## 为什么需要这个页面

现在讲义型文件的切分规则已经基本跑通，但用户入库体验还不完整。主要问题是：

- OCR 完成后，用户不知道下一步应该去哪里看结果。
- 题目可能已经生成了，但“生成草稿”和“正式入库”之间没有明显分界。
- 有些题目只是有小格式问题，不应该和严重识别失败混在一起。
- 用户需要一页页、一题题找问题，批量处理效率低。
- 讲义型文件经常有例题、变式、训练题、解析、水印、页码、题图，结果需要一个集中复核区。

所以，“待入库确认页”不是另一个题库列表，而是一个“入库前工作台”。

## 页面定位

这个页面的定位是：

> 一个批次的 OCR 结果验收页。

它不直接替代“题库核心主库”。核心主库只展示已经确认入库的题目；待入库确认页展示的是“已经识别出来，但还需要用户确认”的题目草稿。

## 推荐入口

建议保留三个入口：

1. OCR 队列卡片里的“查看待入库”
2. 切题批次详情里的“查看识别结果”
3. 题库核心主库顶部的“待入库”入口

最重要的是第一个入口：OCR 完成后，用户应该能从当前批次直接进入，不需要再去题库里找。

## 用户主流程

用户理想路径如下：

1. 上传文件。
2. 系统自动切题。
3. 用户提交切题复核。
4. 系统自动开始 OCR。
5. 系统自动清洗水印、页码和明显格式问题。
6. 系统把结果分成“可入库”和“需处理”。
7. 用户进入待入库确认页。
8. 用户批量确认可入库题目。
9. 用户逐个处理异常题目。
10. 全部处理完成后，批次进入“已完成”。

页面文案上，不要让用户感觉自己在操作一堆任务状态，而是让用户看到：

- 还有多少题可以直接入库。
- 还有多少题需要看一下。
- 哪些题出了什么问题。
- 下一步应该点哪里。

## 页面整体结构

页面建议分成四块：

1. 批次概览
2. 筛选栏
3. 题目列表
4. 右侧预览和操作区

### 1. 批次概览

页面顶部展示这个批次的整体情况。

建议展示：

| 信息 | 展示方式 |
| --- | --- |
| 批次名称 | 文件名或用户设置的资料名 |
| 资料类型 | 试卷型 / 讲义型 / 未确认 |
| 总题数 | 本批次 OCR 得到的题目数量 |
| 可入库 | 没有明显问题，可以批量确认 |
| 需处理 | 有格式、识别、题图、答案解析等问题 |
| 已入库 | 用户已经确认过的题目 |
| 已跳过 | 用户删除或决定不入库的题目 |

建议用一句清楚的状态文案：

- “本批次已识别完成，52 题可直接入库，6 题需要处理。”
- “本批次还有 4 题需要处理，处理完成后即可归档。”
- “本批次已全部处理完成。”

### 2. 筛选栏

筛选栏的目标是让用户快速聚焦问题，不要在长列表里找。

建议提供这些筛选：

- 全部
- 可入库
- 需处理
- 有题图
- 格式问题
- OCR 失败
- 已入库
- 已跳过

排序可以先保持简单：

- 按原文顺序
- 问题优先
- 已处理放后面

第一版可以默认“问题优先”，因为用户最需要先处理异常题。

### 3. 题目列表

左侧列表展示本批次所有题目。每个题目卡片只放必要信息，避免太拥挤。

建议卡片内容：

- 题号或系统编号
- 题型
- 页码范围
- 状态标签
- 是否有题图
- 是否有格式风险
- 题干前 1-2 行预览

状态标签建议使用用户能理解的话：

- 可入库
- 需处理
- 已入库
- 已跳过
- 识别失败

不要使用太内部的状态词，比如 `ready`、`blocked`、`banked` 直接显示给用户。

### 4. 右侧预览和操作区

用户点击某道题后，右侧展示完整内容和操作。

建议右侧分成三段：

1. 原图预览
2. 识别内容
3. 操作按钮

原图预览里需要能看到：

- 整题切片图
- 题干图
- 答案/解析图
- 独立题图

识别内容里展示：

- 题干
- 答案
- 解析
- 知识点
- 解题方法
- 难度

操作按钮建议：

- 确认入库
- 编辑题目
- 重新 OCR
- 格式清洗
- 框选题图
- 跳过此题
- 删除此题

第一层按钮只放最常用的：

- 确认入库
- 编辑题目
- 重新 OCR

其他操作可以放在更多菜单里，避免页面压迫感太强。

## 批量操作

这个页面必须支持批量确认，否则讲义型资料几十道题处理起来会很累。

建议批量操作：

- 全选当前筛选结果
- 批量确认入库
- 批量格式清洗
- 批量重新 OCR
- 批量跳过
- 批量删除

批量确认入库时，建议只默认勾选“可入库”的题目。对“需处理”的题目，除非用户主动勾选，否则不应该被批量入库。

确认按钮文案建议：

- “确认入库 52 题”
- “仅确认当前筛选的 12 题”
- “跳过已选择的 3 题”

如果用户勾选了有风险的题，确认前需要提醒：

> 你选择的题目中有 3 题仍存在格式或识别风险。建议先处理后再入库。

按钮可以提供两个选择：

- 返回处理
- 仍然入库

## 题目状态

页面内部可以按这些状态理解，但展示给用户时用更自然的中文。

| 状态 | 用户看到 | 含义 |
| --- | --- | --- |
| 待确认 | 可入库 | 系统没有发现明显问题，等待用户确认 |
| 需处理 | 需处理 | 有格式、识别、题图或内容风险 |
| 已入库 | 已入库 | 用户确认后进入核心主库 |
| 已跳过 | 已跳过 | 用户决定本题不入库 |
| 识别失败 | 识别失败 | OCR 没拿到可用内容 |

状态变化建议：

- 可入库 → 确认入库 → 已入库
- 可入库 → 跳过 → 已跳过
- 需处理 → 编辑/清洗/重新 OCR → 可入库
- 需处理 → 用户强制确认 → 已入库
- 识别失败 → 重新 OCR → 可入库或需处理
- 识别失败 → 删除/跳过 → 已跳过

## 什么算“需处理”

以下情况建议归入“需处理”：

- OCR 失败或内容为空
- 题干缺失明显
- 答案或解析疑似混入题干
- 有裸露 LaTeX，前端无法正常渲染
- 出现 `ext{C}`、孤立 `\right`、公式定界符不成对
- 水印、页码、版权文字没有被清掉
- 题图数量异常
- 题目跨页但只识别到一部分
- 选择题选项不完整
- 解析很短但原图明显有长解析

这些判断不需要一开始全部做到很智能。第一版可以先依赖现有格式校验和 OCR 状态，后面再逐步加更细的检查。

## 用户文案

页面文案要尽量降低用户负担。

推荐文案：

- “这些题目已经识别完成，确认后会进入题库核心主库。”
- “系统发现以下题目可能需要看一下。”
- “格式清洗只会整理水印、页码和公式显示，不会改写题意。”
- “重新 OCR 会覆盖当前识别内容，请确认后继续。”
- “跳过后本题不会进入题库，可以在本批次记录里查看。”

不推荐文案：

- “bank_status 更新成功”
- “format_review_required = true”
- “OCR draft imported”
- “render error detected”

## 和现有页面的关系

### 和切题复核页

切题复核页只解决“切得对不对”。提交复核后，系统自动进入 OCR。

切题复核页不承担入库确认，不要在这里放太多 OCR 后的编辑功能。

### 和 OCR 队列页

OCR 队列页只解决“有没有跑完、有没有失败”。跑完后，它应该给用户一个明确按钮：

> 查看待入库结果

### 和题库核心主库

核心主库只展示用户已经确认入库的题。不要把大量待处理草稿直接混进去，否则核心主库会越来越像任务列表。

核心主库可以保留一个“待入库”入口，但不应该承担批次级处理。

### 和题目详情页

题目详情页仍然用于单题深度编辑。待入库确认页里的“编辑题目”可以复用当前编辑能力，但用户最好不离开批次上下文。

## 第一版建议做什么

第一版目标是把主流程跑顺，不追求一次做完所有高级筛选。

建议第一版包含：

1. 批次概览
2. 题目列表
3. 状态筛选：全部 / 可入库 / 需处理 / 已入库 / 已跳过
4. 单题预览
5. 单题确认入库
6. 批量确认可入库题
7. 编辑题目
8. 重新 OCR
9. 格式清洗
10. 跳过/删除

第一版可以先不做：

- 很复杂的排序
- 多人协作
- 操作历史回滚
- 高级质量评分
- AI 自动判断题目质量等级

## 第二版可以补什么

第二版可以重点优化效率：

- 只看有题图
- 只看格式错误
- 只看疑似答案解析混排
- 一键处理所有格式问题
- 批量重新 OCR 失败题
- 入库前自动生成知识点和方法建议
- 批次完成报告

批次完成报告可以展示：

- 本批次共处理多少题
- 入库多少题
- 跳过多少题
- 重新 OCR 多少题
- 仍需人工处理多少题

## 需要注意的体验细节

### 不要让用户重复确认

用户在切题复核阶段已经确认过切片，不应该在入库阶段重新面对一堆切题操作。待入库页只在发现问题时让用户介入。

### 不要默认把问题题入库

系统发现明显问题时，可以允许用户强制入库，但不要默认选中。

### 不要把清洗说成“AI 改题”

用户会担心系统改数学内容。文案上要明确：

> 格式清洗只整理显示、噪声和结构，不改题意。

### 保留原图

待入库页一定要能看原图。用户判断 OCR 是否可信，最重要的依据就是原图和识别内容能并排对照。

### 讲义型资料要按批次处理

讲义通常不是一次只有几道题，而是几十道题。页面要支持“批次视角”，不要只做单题视角。

## 推荐页面标题和路径

页面标题：

> 待入库确认

页面副标题：

> 复核 OCR 结果，确认后进入题库核心主库。

推荐路径：

> `/tools/pdf-slicer/runs/:runId/pending-bank`

如果后续不想暴露太长路径，也可以做一个更短的入口：

> `/pending-bank/:runId`

## 完成标准

第一版完成后，用户应该可以完成以下动作：

1. 从 OCR 完成的批次进入待入库确认页。
2. 看到这个批次有多少题可入库、多少题需处理。
3. 快速筛选出需处理题目。
4. 查看单题原图和识别内容。
5. 编辑有问题的题目。
6. 对单题重新 OCR 或格式清洗。
7. 批量确认无问题题目入库。
8. 处理完成后，核心主库只出现已确认入库的题目。

这条链路完成后，用户从“上传讲义”到“确认入库”的路径就会比较完整：

> 上传文件 → 自动切题 → 切题复核 → 自动 OCR → 自动清洗 → 待入库确认 → 核心主库
